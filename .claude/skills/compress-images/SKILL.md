# Compress images for wakon-landing

Images generated by AI tools (or exported straight from a design tool) land in this
project at 1.5-2.5MB per PNG. Referencing them as-is bloats the site and slows the
article/quiz down badly. This skill compresses them to small JPEGs (typically
50-250KB, a ~90% reduction) with no visible quality loss at the sizes these images
are actually displayed at.

## Why JPEG, why quality 82

These are photos and photo-style illustrations (not graphics needing transparency
or crisp flat colors), so JPEG compresses them far better than PNG. Quality 82 with
mozjpeg encoding is the sweet spot this project has used throughout: visually
indistinguishable from the source at normal viewing size, while cutting file size by
90%+. Don't second-guess this default unless the user asks for a specific quality.

## Setup (once per environment)

The compression script needs the `sharp` library. It's a build-time tool, not a
runtime dependency, so install it without touching `package.json`:

```bash
cd <wakon-landing project root>
npm install --no-save sharp
```

If `node -e "require('sharp')"` succeeds, it's already installed — skip this step.

## Running it

The bundled script handles both a single file and a whole folder at once:

```bash
# One file
node .claude/skills/compress-images/scripts/compress.mjs <input> <output> [quality]

# A whole folder — every .png/.jpg/.jpeg/.webp inside gets compressed to .jpg
node .claude/skills/compress-images/scripts/compress.mjs <input-dir> <output-dir> [quality]
```

It prints a before/after size table (and a TOTAL row in batch mode) — always share
that with the user so they can see the win, the way file sizes dropped from ~1.8MB to
~100KB is the whole point of doing this.

## Where things go in this project

- **Article images** (the advertorial in `src/articleContent.es.js` / `.en.js`):
  destination is `public/shared/images/article/<name>.jpg`. The filename must match
  whatever `heroImage`, `discoveryImage`, `scienceImage`, etc. point to in those two
  content files — check both, they share the same image paths.
- **Quiz / shared images** (testimonials, avatars, social-proof, author photo):
  destination is `public/shared/images/<name>.jpg`.
- **App icons or anything else**: ask the user, or infer from where the existing
  file it's replacing lives.

If the user hands you a folder of newly generated images with descriptive filenames
that don't exactly match the destination slot names (e.g. `hilos gold y azul.png`
meant for `science.jpg`), don't just batch-copy blindly — look at each image (the
`Read` tool can view images directly) and match it to the right destination based on
what it actually shows and the surrounding article copy, the same way you'd match a
photo to a caption. Compress each one individually with the correct destination
filename rather than running batch mode with mismatched names.

## After compressing

1. Confirm the files landed at the right destination paths with the right filenames.
2. If this is a genuinely new image (not replacing an existing reference), you still
   need to add the path to the relevant content file — compressing alone doesn't wire
   it in.
3. Start/reuse a preview server and check the image actually renders where expected
   (existing site conventions: `mcp__Claude_Preview__preview_start` with the
   `wakon-landing` config, then look at the relevant screen). Don't just trust the
   file exists — confirm it loads with no broken paths.
4. Clean up: this skill's script is the only file that should exist — never leave a
   stray one-off `.mjs` conversion script lying around in the repo root after you're
   done (that used to happen before this skill existed; now there's no reason to
   write throwaway scripts by hand).
